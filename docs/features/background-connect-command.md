# Background Connect Command

## Goal

CLI tokens generated by `internal/hub/server.go:handleCreateBridgeToken` should
produce a connect command that installs Bridge as a restartable background
service when the user's machine supports `systemd --user`. The command should
return the terminal immediately and survive normal reconnects and reboots.

## Non-Goals

- Change the `/install.sh` download behavior.
- Change the Bridge WebSocket protocol, enroll token semantics, or runner
  behavior.

## Current State

The settings UI displays commands returned by `POST /api/bridge-tokens`. Before
this change, the connect command ran `codex-bridge connect` in the foreground.
That is technically correct because Bridge is a long-lived reverse WebSocket
client, but it makes the copied command feel like it hangs after successful
connection.

## Design

`internal/hub/server.go:bridgeConnectCommand` generates a shell command that:

- Computes `CB_CWD`, `CB_DIR`, and `CB_HASH` from the current directory.
- Creates `~/.codex-bridge/machines`, `~/.codex-bridge/logs`, and
  `~/.codex-bridge/services`.
- If `systemctl --user` is available, writes
  `~/.config/systemd/user/codex-bridge-<cwd-hash>.service` and a matching
  start script, then runs `systemctl --user enable` and `restart`.
- Best-effort enables lingering with `loginctl enable-linger` so the user
  service can start after reboot without an interactive login.
- Falls back to `nohup ... &` when user systemd is unavailable.
- Prints the service name or background PID plus log path.

The command keeps the existing per-directory machine id file:
`$HOME/.codex-bridge/machines/${CB_HASH}`. Existing enroll token rules remain:
a token can reconnect the same machine id, but a different machine id requires a
new token.

## Reviewer Q&A

**Q: Why install a user service instead of only using `nohup`?**

`nohup` does not survive a machine reboot. A user service gives automatic
restart behavior without needing root-level service files.

**Q: What if the target machine does not have user systemd?**

The generated command falls back to the previous `nohup` behavior and prints the
background PID.

**Q: Where does stdout/stderr go?**

The command writes logs to `~/.codex-bridge/logs/<cwd-hash>.log` and prints that
path after startup.
