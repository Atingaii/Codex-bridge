# Markdown Message Rendering

## Goals

Render common Markdown blocks in chat, orchestration, and public transcript
messages through the shared `frontend/src/app/components/chat/MessageContent.tsx`
component. In particular, GitHub-style pipe tables such as:

```markdown
| Path | Purpose |
| --- | --- |
```

must display as tables instead of plain text with `<br>` separators.

The renderer should also cover the common formats produced by Codex and Claude:
headings, paragraphs, unordered and ordered lists, blockquotes, horizontal
rules, fenced code blocks, inline code, strong/emphasis, links, and safe image
Markdown.

## Non-Goals

- Do not change HTTP endpoints, WebSocket frames, SQLite schema, runners, auth,
  or continuity behavior.
- Do not execute or render arbitrary HTML from model output.
- Do not add a server-side Markdown rendering service.
- Do not make a full CommonMark-compatible parser in this pass.

## Data And Protocol Impact

None. Persisted message content remains plain text Markdown in
`internal/store/store.go:Message`. The change is display-only in
`frontend/src/app/components/chat/MessageContent.tsx:MessageContent` and shared
helpers in `frontend/src/app/lib/utils.ts`.

## Design

Keep rendering local to the browser and preserve the existing safety boundary:
escape raw message text first, then add a constrained HTML subset generated by
the renderer. Raw HTML remains escaped and visible as text.

`frontend/src/app/lib/utils.ts:renderMarkdown` renders block-level Markdown.
`frontend/src/app/lib/utils.ts:renderInlineMarkdown` remains the shared inline
formatter and is expanded to support emphasis, links, and safe images. The
message component continues using `dangerouslySetInnerHTML`, but only receives
HTML emitted by these escaping helpers.

Tables are detected as a header row followed by a delimiter row. Delimiter cells
may use `---`, `:---`, `---:`, or `:---:` for left, right, or center alignment.
Rows are split on unescaped pipe characters so inline code such as `` `a|b` ``
does not create extra columns.

## Implementation Steps

1. Add this feature document before code changes.
2. Replace the message component's newline-to-`<br>` path with the shared block
   Markdown renderer.
3. Add CSS classes through Tailwind utility class strings emitted by the
   renderer so tables and other blocks match the existing chat card density.
4. Rebuild `internal/web/static/` with `npm run build` from `frontend/`.
5. Run focused and broad verification gates.

## Exit Gates

- Pipe tables render with `<table>`, `<thead>`, `<tbody>`, `<th>`, and `<td>`.
- Markdown headings, lists, blockquotes, horizontal rules, fenced code blocks,
  inline code, bold, emphasis, links, and safe image syntax render correctly.
- Raw HTML in model output is escaped instead of executed.
- Chat, orchestration, and public share/snapshot transcripts keep using the
  shared message renderer.
- `npm run build` refreshes `internal/web/static/`.
- `/usr/local/go/bin/go test ./...`, `make doc-lint`, and `git diff --check`
  pass.

## Reviewer Q&A

**Q: Why not add a Markdown package?**

A: The current app has no Markdown dependency or sanitizer. A small constrained
renderer keeps the security model explicit and avoids rendering arbitrary HTML.

**Q: Does this alter stored conversations?**

A: No. Messages stay stored as Markdown/plain text. Only browser presentation
changes.

**Q: Does this affect follow-up continuity?**

A: No. Chat still reuses the same `sid` and orchestration follow-ups still use
`POST /api/orchestrations/{runID}/prompts`.
